# Backend File Structure & Contribution Guide

This document explains the backend project layout, what belongs where, and how to add new files or apps. It’s written for the Nascent-NepalUS hackathon MVP backend (Django + DRF). Use it as a single-source reference when extending or contributing to the backend.

Root paths referenced here use the repository root `Nascent-NepalUS/` and the Django project root `backend/`.

---

## Top-level layout (summary)

- `backend/` — Django project (project settings, top-level urls, WSGI/ASGI)
- `core/` — shared utilities, constants, small helpers used across apps
- `users/` — user profile, registration, authentication-related models and endpoints
- `tasks/` — Task planner, create/list/update/complete tasks
- `focus/` — Focus sessions / Pomodoro session tracking
- `sleep/` — Sleep entries (manual or inferred)
- `analytics/` — Burnout calculation, history, and any analytics services
- `requirements.txt`, `Dockerfile`, `docker-compose.yml`, `.env.example` — infra & env examples
- `manage.py` — standard Django manage script

---

## What to place where (detailed)

### `backend/`
- `backend/settings.py` — central settings. Place environment-driven configuration here. Add `BURNOUT_WEIGHTS` and threshold constants here (so teams can tune them).
- `backend/urls.py` — top-level URL patterns. Wire the app routers here (or include `core`/app urls).
- `backend/wsgi.py` / `backend/asgi.py` — deployment entrypoints.
- Task for you: when you add a new app, import/register its router or path from here.

### `core/`
- `core/utils.py` — small helpers (date ranges, normalization helpers).
- `core/constants.py` — shared constants, e.g., score buckets, action strings.
- `core/permissions.py` — centralized DRF permissions if you add custom ones.
- Task: put generic helpers here (do not add domain-specific logic).

### `users/`
- `users/models.py` — `UserProfile` or extensions to `AUTH_USER_MODEL`.
- `users/serializers.py` — registration and profile serializers.
- `users/views.py` — auth endpoints (or delegate to `rest_framework_simplejwt`).
- Task: keep auth and profile concerns here only.

### `tasks/`
- `tasks/models.py` — `Task` model (title, type, completed, timestamps).
- `tasks/serializers.py` — create/update/list serializers.
- `tasks/views.py` — DRF `ViewSet` or API views to manage tasks.
- `tasks/urls.py` (optional) — app-specific router if you prefer.
- Task: task creation, completion toggles, filtering by `task_type` live here.

### `focus/`
- `focus/models.py` — `FocusSession` model (duration_seconds, timestamp).
- `focus/serializers.py` — serializer for sessions.
- `focus/views.py` — endpoints to start/record/list sessions.
- Task: use this app to store PC/mobile-reported focus sessions and simple analytics.

### `sleep/`
- `sleep/models.py` — `SleepEntry` (date, hours, optional quality).
- `sleep/serializers.py` — input validation (hours must be reasonable).
- `sleep/views.py` — create/list sleep entries.
- Task: store manual sleep inputs; keep logic simple (no ML).

### `analytics/`
- `analytics/models.py` — `BurnoutScore` history (score, level, suggested_action, timestamp).
- `analytics/services.py` — central rule-based burnout calculation logic (normalize inputs, apply weights, map to action). This file should query `tasks`, `sleep`, and `focus` models or accept injected data for computation.
- `analytics/views.py` — endpoint(s) that return the current burnout score and a history endpoint.
- `analytics/serializers.py` — serializer for `BurnoutScore`.
- Task: all burnout calculation logic and score-history persistence lives here. Keep it deterministic and settings-driven.

## File-level responsibilities & naming conventions

- Models: `models.py` per app with model classes using singular names (e.g., `Task`, `FocusSession`).
- Serializers: `serializers.py` per app. Use `ModelSerializer` where possible.
- Views: `views.py` per app. Prefer `ViewSet` for standard CRUD.
- URLs: either `urls.py` per app or register DRF routers in `backend/urls.py`.
- Services/Business Logic: `services.py` for heavy domain logic (e.g., `analytics/services.py`). Keep views thin.
- Tests: `tests/` package or `tests.py` per app. Add unit tests for `services.py` and model methods.
- Migrations: `migrations/` auto-generated by Django. Commit them.

## How to add a new app (step-by-step)

1. Create the app:
   - Run `python manage.py startapp <appname>` (from inside `backend/` project).
   - This creates `<appname>/` with standard files.

2. Register the app:
   - Add `<appname>` to `backend/settings.py` `INSTALLED_APPS`.

3. Implement models:
   - Add your models to `<appname>/models.py`.
   - Use `ForeignKey(settings.AUTH_USER_MODEL, ...)` for user relationships.

4. Create serializers & views:
   - Add `<appname>/serializers.py` and `<appname>/views.py`. Prefer DRF `ModelViewSet` for CRUD.

5. Add URLs:
   - Option A: add a `DefaultRouter` entry in `backend/urls.py`:
     - e.g., `router.register(r'<appname>/', <AppName>ViewSet, basename='<appname>')`
   - Option B: add `include('<appname>.urls')` if you prefer app-local routing.

6. Make & run migrations:
   - `python manage.py makemigrations <appname>`
   - `python manage.py migrate`

7. Add tests:
   - Add unit tests to `<appname>/tests/`.
   - Ensure tests run locally: `pytest` or `python manage.py test`.

8. Document:
   - Add or update `FILE_STRUCTURE.md` (this file) and add app-specific README if complex.

9. Commit & PR:
   - Follow contribution guidelines below before opening a pull request.

When you add a `ViewSet` in `newapp/views.py`, register it in `backend/urls.py` router:

- Add import and registration (conceptual inline example):
  - `router.register(r'newapp', NewAppViewSet, basename='newapp')`

(Keep router registration concise and consistent with other apps.)

## Burnout calculation: placement & guidance

- Put the core rule-based logic in `analytics/services.py`.
- Keep the service pure where possible:
  - Accept normalized inputs or accept a `user` and `date` and query `sleep`, `tasks`, and `focus`.
- Store historical runs in `analytics/models.py` as `BurnoutScore` so the mobile app can show trends.
- Keep weights and thresholds in `backend/settings.py` so they can be tuned without code changes.

## Tests & QA

- Unit test:
  - Test model behavior (`Task.mark_complete()`), serializer validation, and service outputs (burnout score normalization and mapping).
- Integration:
  - Test endpoints for expected responses and permission behavior.
- Add test fixtures for common scenarios: healthy user, moderate burnout, high burnout.

## Contribution guidelines (quick)

- Branch naming: `feature/<short-description>`, `fix/<short-description>`, or `chore/<short>`.
- Commits:
  - Write concise, present-tense messages. Include an issue reference if present.
- Pull Requests:
  - PR description should include intent, screenshots (if relevant), and migration notes.
  - Assign reviewers and link to the hackathon task card.
- Linting & formatting:
  - Run `black` and `flake8` (or your chosen linters) before opening a PR.
- Tests:
  - Include tests for new functionality. All tests should pass on CI before merging.
- Review:
  - Keep changes small and focused. If a change touches multiple apps, split into multiple PRs if possible.

## Environment & local dev tips

- Use `.env` or `.env.local` to hold secrets. Example keys live in `.env.example`.
- For quick local demo use SQLite; for production/demo via Docker use Postgres in `docker-compose.yml`.
- To run locally:
  - `pip install -r requirements.txt`
  - `cd backend`
  - `python manage.py migrate`
  - `python manage.py createsuperuser` (optional)
  - `python manage.py runserver`

## Adding new files (best practices)

- Keep files focused: each file should have a clear responsibility (models, views, serializers).
- Name files predictably: `models.py`, `views.py`, `serializers.py`, `services.py`.
- Add module docstrings or a short header comment describing purpose.
- Update `backend/urls.py` and `README.md` when adding new public endpoints.
- Add tests in the same app so behavior is easy to reason about.

## Helpful quick-check checklist before PR

- [ ] Did you add the app to `INSTALLED_APPS`?
- [ ] Did you create and commit migrations?
- [ ] Are new endpoints wired in `backend/urls.py` or app `urls.py`?
- [ ] Are new settings variables documented in `backend/settings.py` (and `.env.example` if needed)?
- [ ] Did you add unit tests for service logic?
- [ ] Did you run `black` and linter checks?
- [ ] Is the PR description clear about what changed and why?
